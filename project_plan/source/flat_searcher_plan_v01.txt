# Full Product Plan: Riga Apartment Analyzer for SS.com

## 1. Product concept

The project is a Python desktop application for analyzing apartments for sale in Riga.

The first supported source is SS.com, specifically the Riga apartment sale section:

`https://www.ss.com/lv/real-estate/flats/riga/all/sell/`

The application collects apartment listings, parses their details, analyzes the listing text and images using AI, enriches listings with geodata, calculates different scores, and shows apartments in two main views:

1. Ranked apartment list.
2. Interactive Riga map with apartment markers and clusters.

The application is not just a scraper or map viewer. The core value is turning chaotic apartment listings into structured apartment candidates with explainable ranking.

The main user scenario is finding a good apartment for living, with a preference for mortgage suitability, reasonable price/value, usable private rooms, acceptable location, and basic infrastructure nearby.

The default profile is:

`For living + mortgage`

## 2. Core product principle

The system should not trust SS.com fields blindly.

SS.com table fields are useful as initial signals: price, area, declared rooms, floor, district, street, building series, and link.

The main source of deeper interpretation is:

1. Listing text.
2. Listing images.
3. AI interpretation.
4. Optional layout priors based on building series/type.
5. Geodata and market comparison.

The application should evaluate the real usefulness of the apartment, not only what the seller claims.

Important examples:

1. SS may say ŌĆ£2 roomsŌĆØ, but AI may identify ŌĆ£1 private room + kitchen-livingŌĆØ.
2. SS may say ŌĆ£3 roomsŌĆØ, but AI may identify ŌĆ£2 private rooms + 1 walkthrough/common roomŌĆØ.
3. SS may provide only a street without a house number, in which case geolocation is approximate and location-sensitive scores are not calculated.
4. A cheap apartment may be a good opportunity, but if it is suspiciously cheaper than similar apartments, the app should show a warning.
5. A good apartment may be a poor mortgage object, so bankability is a separate scoring block and warning layer.

## 5. Caching and update logic

The application should not reanalyze everything on every launch.

On first launch:

1. Open SS.com Riga apartment sale list.
2. Collect all active listings.
3. Open each detailed listing page.
4. Store raw listing data in local database.
5. Download images temporarily for AI analysis.
6. Run AI analysis.
7. Store structured AI output, confidence labels, explanations, flags and scores.
8. Remove temporary non-plan images unless needed for debugging.
9. Keep only necessary metadata, AI conclusions, original listing URL, text and floor plan image if found.

On later launches:

1. Recheck the SS.com listing page.
2. Detect new listings.
3. Detect removed/inactive listings.
4. Detect changed price.
5. Detect changed listing text.
6. Detect changed image count.
7. Detect changed unique visit count.
8. Run full AI analysis only for new listings or listings that changed enough to require reanalysis.
9. Recalculate scores.
10. Preserve history.

Images should generally not be stored permanently. Ordinary photos are used by AI internally and then removed. The exception is a floor plan image, which may be saved or cached because it directly explains the layout conclusion in the UI.

## 6. Image display policy

The apartment detail view should not show ordinary listing photos.

Do not show:

1. Room photos.
2. Kitchen photos.
3. Bathroom photos.
4. Facade photos.

The only image type that may be shown to the user is:

`floor plan image`

Reason: the floor plan directly explains the layout conclusion.

The apartment detail view should include a link to the original SS listing, where the user can view all original photos.

## 10. Two-pass Gemini AI pipeline

The AI analysis uses a two-pass Gemini pipeline.

### 10.1 Pass 1: image classification and grouping

The first AI pass must not aggressively discard images.

Interior photos are often important because layout can be inferred from small clues: floors, wallpaper, windows, doors, lighting, furniture, repeated corners, hallway transitions and room continuity.

Pass 1 tasks:

1. Classify all images.
2. Find floor plan images.
3. Mark useless images.
4. Detect near-duplicates.
5. Group interior photos by likely room.
6. Detect corridor/hallway/entrance photos.
7. Detect kitchen, bathroom and living room photos.
8. Mark exterior/building photos.
9. Identify photos useful for layout reasoning.
10. Identify photos useful for building/mortgage analysis.

Image categories:

1. `floor_plan`
2. `interior_room`
3. `kitchen`
4. `bathroom`
5. `corridor_or_hallway`
6. `exterior_building`
7. `entrance_staircase`
8. `yard_or_street_view`
9. `duplicate_or_near_duplicate`
10. `irrelevant_or_decorative`
11. `agency_collage`

Almost all useful interior photos should be passed to the second AI pass unless they are obvious duplicates or irrelevant.

### 10.2 Pass 2: layout, mortgage and explanation reasoning

The second AI pass receives:

1. Full listing text.
2. SS table fields.
3. Useful interior photos.
4. Floor plan image if found.
5. Pass 1 image classification.
6. Image groups by likely room.
7. Exterior/building photos for building risk analysis.
8. Optional layout priors if available.

Pass 2 determines:

1. Real layout.
2. Effective private rooms.
3. Walkthrough rooms.
4. Kitchen-living cases.
5. SS vs AI room conflict.
6. Building type guess.
7. Series guess.
8. Wooden building risk.
9. Stove heating risk.
10. Mortgage risk level.
11. Human-readable explanations.

## 11. Floor plan rule

If a floor plan image is found, it becomes the primary source for layout analysis.

Evidence priority:

1. Floor plan image.
2. Interior photos as cross-check.
3. Listing text.
4. SS-declared room count.
5. Typical layout prior.

Interior photos should be used to check:

1. Whether photos match the plan.
2. Whether the plan likely belongs to the same apartment.
3. Whether room count is confirmed.
4. Whether walkthrough areas are visible.
5. Whether kitchen/corridor/rooms match the plan.

There is no separate user-facing `floor plan found` flag. The presence of the plan is visible through the layout explanation and the displayed floor plan image.

## 12. Exterior photo use

Facade and exterior photos are not used directly to determine room layout.

They are used for:

1. Building type guess.
2. Series guess.
3. Wooden building risk.
4. Facade condition.
5. Renovation or insulation signs.
6. Mortgage risk.
7. Liquidity risk.
8. Building condition context.

## 14. Product rules for room classification

### 14.1 Private room

A private room is a living room that can be entered from a neutral area, such as:

1. Corridor.
2. Hallway.
3. Entrance hall.
4. Common non-private area.

It may be:

1. Bedroom.
2. Living room.
3. ChildrenŌĆÖs room.
4. Office.
5. Other usable living room.

The key rule: a person should not need to pass through another personŌĆÖs private room to access their own room.

### 14.2 Walkthrough room

A walkthrough room is a living room that must be crossed to enter another living room.

For rating, walkthrough rooms are not counted as private rooms.

### 14.3 Kitchen

Kitchen is not counted as a living room.

Kitchen quality is not evaluated as a separate factor. Kitchen size, shape or convenience does not matter unless it affects classification as kitchen-living.

### 14.4 Kitchen-living / studio

Kitchen-living is not counted as a full private room.

Product rule:

`1 bedroom + kitchen-living ŌēĀ good 2-room apartment`

It should be interpreted as:

`1 private room + common kitchen-living area`

### 14.5 Bathroom and kitchen access

Do not complicate the model by analyzing detailed access paths to bathroom or kitchen.

The main layout logic is:

1. How many usable private living rooms exist.
2. Whether any living rooms are walkthrough.
3. Whether kitchen-living is being presented as a room.

## 15. Layout scoring

The default target is:

`effective_private_rooms = 2`

Layout scoring rules:

1. Two effective private rooms are the target case.
2. Three or more effective private rooms are good, but not a huge bonus.
3. One effective private room is weaker for the default profile, but the listing remains visible.
4. Unclear layout does not receive strong layout bonus, but should not collapse the overall score.
5. SS vs AI conflict is shown as a warning.
6. Kitchen-living plus one bedroom is not treated as a good two-room apartment.
7. Room count influences score, but should not dominate score.

The number of rooms should be configurable through profile importance. In the default profile, room privacy matters, but price/value and mortgage suitability remain highly important.

## 16. Mortgage suitability / bankability

The application should calculate a separate mortgage suitability score.

A good apartment and a good mortgage object are not necessarily the same thing.

Mortgage suitability should be represented by:

1. `mortgage_bankability_score`
2. `mortgage_risk_level`
3. `mortgage_risk_reasons`

In the default profile `For living + mortgage`, mortgage risk affects ranking but does not automatically remove the listing.

High-risk listings remain visible but receive visible warnings.

In the profile `Mortgage first`, mortgage risk has much stronger ranking impact.

### 16.1 Risk factors

Factors that reduce mortgage suitability:

1. Stove heating.
2. Wooden building.
3. Wooden building plus stove heating.
4. Legal risks.
5. Land lease.
6. Shared ownership.
7. Encumbrances.
8. Auction.
9. Building not commissioned.
10. Poor building condition.
11. Poor facade condition.
12. Low liquidity.
13. Missing basic utilities.
14. No normal bathroom/water/sewer infrastructure.
15. Very incomplete or suspicious listing.

Stove heating:

`High` or `Critical`

Wooden building:

`High`

`Critical`

Legal/ownership complications:

`High` or `Critical`

2. Are lowered by the mortgage suitability block.
3. Receive a clear warning flag.
4. Can be hidden by filters if the user chooses.

## 17. Price-value scoring

Price-value is the central scoring block in the default profile.

There is no default budget.

Do not define soft limits such as:

1. Up to 100k.
2. Up to 120k.
3. Above 130k is bad.

The default principle:

`Price/value is more important than absolute cheapness.`

A more expensive apartment may rank above a cheaper one if it has better Ōé¼/m┬▓, better usable area, better location, better private rooms and stronger overall usefulness.

### 17.1 Components of price_value_score

The price-value score consists of:

1. `price_per_m2_score`
2. `relative_market_score`
3. `price_per_effective_private_room_score`
4. `absolute_price_score`
5. `suspicious_low_price_flag`

### 17.2 Price per m┬▓

`price_per_m2_score` is the main default price factor.

It compares the apartmentŌĆÖs Ōé¼/m┬▓ to the market and similar listings.

### 17.3 Relative market score

`relative_market_score` compares the listing to similar apartments.

If the apartment is cheaper than comparable listings, it receives a plus.

If it is more expensive, it receives less value or a negative adjustment.

### 17.4 Price per effective private room

Internal metric:

`price_per_effective_private_room = price_eur / effective_private_rooms`

Use this only when AI confidently identified effective private rooms.

This metric is important for cases such as:

1. `SS: 3 rooms, AI: 2 private + walkthrough`
2. `SS: 2 rooms, AI: 1 private + kitchen-living`
3. `SS: 2 rooms, AI: unclear`

If effective rooms are unclear, do not calculate this metric to avoid false penalty.

### 17.5 Absolute price score

Absolute price is a weak default factor.

It can be strengthened in a custom profile if the user wants cheaper apartments to rank higher.

### 17.6 Suspiciously low price

If a listing is far cheaper than similar apartments, it receives both:

1. Positive price-value contribution.
2. Warning flag: `Suspiciously low price ŌĆö check carefully`

Cheapness should not kill the rating. It should create a warning when it looks abnormal.

### 17.7 Price history

Price history does not affect rating.

The following are context only:

1. Price drop.
2. Price increase.
3. View growth.
4. Listing text changed.
5. Photo count changed.
6. Listing disappeared and returned.

They are shown in the detailed view, not used in score.

## 18. Relative market comparison

The system needs market baselines to decide whether a price is good.

Use multiple baseline levels.

### 18.1 Level 1: Riga baseline

Always calculate a broad Riga baseline.

Use:

1. Active Riga apartment listings.
2. Broad property type match.
3. Outlier filtering.

This ensures that price scoring works even when there are not enough close comparables.

### 18.2 Level 2: district baseline

Compare with listings in the same district.

Suggested matching:

1. Same district.
2. Area ┬▒20%.
3. SS rooms ┬▒1.

### 18.3 Level 3: AI-adjusted baseline

If AI confidently identified effective private rooms:

1. Same effective private room count.
2. Same district.
3. Area ┬▒20%.
4. Same or similar building type if available.

### 18.4 Level 4: series/building baseline

If building series/type is known:

1. Same series, or similar building type.
2. Area ┬▒20%.
3. Same district or nearby districts.

### 18.5 Exclusions from normal market baseline

Critical-risk listings should not distort normal market baselines for the default profile.

Exclude from normal baseline:

1. Critical mortgage risk.
2. Obvious stove heating plus wooden building.
3. Severe legal risk.
4. Inactive listings.
5. Very incomplete listings.
6. Extreme price/area outliers.
7. Obvious data errors.

These listings may still appear in the ranking if they pass filters. They are simply not used as normal comparables.

### 18.6 Outlier handling

Use median, percentiles or trimmed mean rather than simple average.

Outlier examples:

1. Clearly wrong price.
2. Clearly wrong area.
3. Unrealistic Ōé¼/m┬▓.
4. Share/auction/partial ownership listings.
5. Severely incomplete listings.

### 20.1 Price value

Evaluates:

3. Relative market value.
4. Price per effective private room.
5. Price compared with similar apartments.

### 20.2 Useful area

Evaluates area comparatively.

There is no hard lower bound.

Rules:

1. More area is good if price does not rise disproportionately.
2. Smaller area is not automatically bad if price and location are good.
3. Area should be evaluated together with price, Ōé¼/m┬▓ and layout.

### 20.3 Room privacy

Evaluates:

1. Effective private rooms.
2. Walkthrough rooms.
3. Kitchen-living cases.
4. SS vs AI room mismatch.

### 20.4 Layout confidence

Evaluates confidence in AI layout conclusion.

Uses:

1. Floor plan presence.
2. Layout evidence.
3. AI confidence label.
4. SS vs AI conflict.
5. Uncertainty reason.

Low confidence should create warning, but not destroy score.

### 20.5 Mortgage suitability

Evaluates bankability of the object.

Risks include stove heating, wooden building, legal issues, land lease, encumbrances, auction, building status and poor building condition.

### 20.6 RTU accessibility

For MVP, this is based on distance to one main RTU point.

No travel time or route planning in MVP.

### 20.7 Transport connectivity

For MVP, this is based on proximity and count of nearby stops if available.

No route planning, intervals, schedules or travel time in MVP.

### 20.8 Central station accessibility

For MVP, this is based on distance to R─½ga Centr─ül─ü / Origo area.

### 20.9 Shops / infrastructure

For MVP, this is based on nearby grocery shops.

### 20.10 Building / series

Evaluates:

1. Building series.
2. Building type.
3. Material.
4. Typical layout risk.
5. Facade condition.
6. Entrance/staircase condition.
7. Potential mortgage/liquidity risks.

### 20.11 Floor

Disabled in default profile.

Can be enabled in custom profiles.

Potential custom settings:

1. Penalize first floor.
2. Penalize top floor.
3. Prefer middle floors.
4. Consider elevator.

### 20.12 Condition / renovation

## 21. Score calculation logic

Use a hybrid scoring model.

Step 1: each scoring block returns a 0ŌĆō100 score.

Step 2: profile importance converts into weights.

Suggested weights:

1. Ignore = 0.
2. Weak factor = 1.
3. Medium factor = 2.
4. Strong factor = 3.
5. Critical factor = 5.

Step 3: calculate weighted score.

Step 4: apply penalties where appropriate.

Step 5: apply tie-breakers when apartments are close.

Formula concept:

`overall_score = weighted_average(block_scores) - penalties`

### 21.1 Tie-breakers

If two apartments have similar scores, for example within 3ŌĆō5 points, apply tie-breakers.

Default profile tie-breakers:

1. Better price value.
2. Better location / RTU / transport.
3. Better mortgage suitability.
4. Better confirmed room privacy.
5. More useful area at similar price.

This supports the product rule that two apartments can stand near each other, but the cheaper and better-located one may be slightly higher.

## 26. Geocoding and address precision

Location-sensitive scores are calculated only for exact addresses.

Exact address means:

`street + house number`

Address precision levels:

1. `exact_house`
2. `street_approx`
3. `district_approx`
4. `unknown`

### 26.1 exact_house

Requirements:

1. Street is known.
2. House number is known.
3. Geocode confidence is high.

Behavior:

1. Show normal map marker.
2. Calculate RTU score.
3. Calculate central station score.
4. Calculate shop score.
5. Calculate transport score if transport block is enabled.

### 26.2 street_approx

Street is known, but house number is missing.

Behavior:

1. Show apartment on map.
2. Use approximate street-level coordinate, such as street center or best geocoder result.
3. Marker must look visually different, for example transparent, dashed or with `Ōēł`.
4. Do not calculate location-sensitive scores.
5. Show flag: `Approximate address ŌĆö location scores not calculated`.

### 26.3 district_approx

Only district is known.

Behavior:

1. May show in district center with strong warning.
2. Use a more approximate marker than street-level marker.
3. Do not calculate location-sensitive scores.

### 26.4 unknown

No usable coordinates.

Behavior:

1. Do not show on map.
2. Keep in ranking if it passes filters.
3. Location-sensitive scores are zero/not awarded.

### 26.5 Geocode confidence

Add confidence levels:

1. `high`
2. `medium`
3. `low`

For MVP scoring:

`exact_house + high confidence` means scores can be calculated.

All other cases mean location scores are not calculated.

This avoids false location bonuses from bad geocoding.

## 27. Location scoring

For MVP, calculate distances, not routes or travel time.

Do not calculate:

1. Walking time.
2. Public transport travel time.
3. Route planning.
4. Transfers.
5. Frequency.
6. Schedules.
7. ŌĆ£Can reach in 10 minutesŌĆØ.

### 27.1 RTU-score

Use one main RTU point.

`rtu_destination_mode = single_main_point`

Score by distance only.

Example scale:

1. Up to 1 km: very strong plus.
2. 1ŌĆō2 km: strong plus.
3. 2ŌĆō4 km: normal plus.
4. 4ŌĆō7 km: weak plus.
5. Further: almost zero.

Use a smooth score, not a hard filter.

### 27.2 Station-score

Use R─½ga Centr─ül─ü / Origo as the central station destination.

Treat them as one central zone or one destination point.

`station_destination_mode = central_station_zone`

Score by distance only.

Example scale:

1. Up to 1 km: very strong plus.
2. 1ŌĆō3 km: strong plus.
3. 3ŌĆō6 km: normal plus.
4. 6ŌĆō10 km: weak plus.
5. Further: almost zero.

### 27.3 Shop-score

For MVP, count grocery stores.

Examples:

1. Rimi.
2. Maxima.
3. Lidl.
4. Mego.
5. Elvi.
6. Top.
7. Beta.
8. Mere.
9. Other grocery/supermarket objects.

Shopping centers are not a separate main scoring layer for MVP.

If a grocery store inside a mall appears as a separate map object, count it.

Later, a mall may provide weak infrastructure score if known to contain groceries.

Suggested grocery radii:

1. Up to 300 m: strong plus.
2. 300ŌĆō700 m: normal plus.
3. 700ŌĆō1200 m: weak plus.
4. Further: zero.

Also count number of shops nearby:

1. One shop within 300 m is good.
2. Several shops within 700 m is better.
3. Many shops nearby gives high shop score.

### 27.4 Transport-score

For MVP, keep transport simple.

Use:

1. Nearby stop distance.
2. Number of nearby stops.
3. Number of routes only if easy to obtain.

Do not use:

1. Route planning.
2. Schedules.
3. Frequency.
4. Intervals.
5. Travel time to RTU.
6. Travel time to station.

If address is not exact and high-confidence, transport score is not calculated.

## 32. Layout prior database

The application may use a local database of typical building layouts.

Gemini does not directly query the database. The application retrieves relevant prior layouts and includes a small number of candidates in the prompt.

Pipeline:

1. Extract listing features.
2. Search local layout prior database.
3. Retrieve 3ŌĆō10 relevant candidates.
4. Pass them to Gemini as hypotheses.
5. Gemini uses them as supporting context, not as truth.

Fields for layout prior:

1. `series_name`
2. `building_type`
3. `construction_period`
4. `typical_area_range`
5. `typical_room_count`
6. `typical_layout_variants`
7. `walkthrough_probability`
8. `isolated_rooms_probability`
9. `source_url_or_note`
10. `confidence`
11. `example_floorplans_optional`
12. `created_by`
13. `verified`

Important rule:

Typical layout prior cannot override real evidence.

If a typical Khrushchevka layout usually has walkthrough rooms, but the floor plan clearly shows two separate rooms with separate corridor access, the floor plan wins.

## 35. Final product summary

The application analyzes Riga apartment listings from SS.com and turns them into structured, explainable apartment candidates.

It provides:

1. AI-derived effective private room count.
2. SS vs AI room conflict detection.
3. Mortgage risk assessment.
4. Price-value score.
5. Relative market comparison.
6. Location scores only when address is exact.
7. Map with exact and approximate markers.
8. Clustered apartment view.
9. User statuses for real search workflow.
10. Filters independent from ranking.
11. Scoring profiles and custom profile editor.
12. Clean English UI without technical AI/debug details.
13. Original SS listing link and source text.
14. Listing history for price, visits and changes.

The main product rule:

The app should help the user find good apartment opportunities without hiding uncertainty.

A listing can be attractive but uncertain. A listing can be cheap but risky. A listing can have good SS room count but poor AI-effective room count. The application should not conceal these conflicts; it should rank, explain and warn clearly.
